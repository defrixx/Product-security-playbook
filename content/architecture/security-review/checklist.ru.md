# Чеклист ревью архитектуры безопасности

## 1. Область, цель и контекст

Этот чеклист нужен для того, чтобы системно оценить, как архитектурные изменения влияют на риск безопасности до релиза.

Фокус ревью:
- выявить новые или усиленные векторы атак;
- проверить, что ключевые контроли безопасности действительно работают для целевых сценариев;
- зафиксировать остаточные риски, владельцев и условия принятия решения по релизу.

### Входные артефакты

- актуальная архитектурная диаграмма;
- описание набора изменений (PR/тикет/ADR);
- описание потоков данных и границ доверия;
- список владельцев компонентов;
- policy/конфиги и результаты релевантных тестов/сканов.

### Выходы ревью

- список findings в стандартизированном формате (раздел 5);
- зафиксированные решения/компромиссы и принятые риски (раздел 5);
- финальный security-вердикт по правилам gate (раздел 7).

Трассируемость обязательна: каждый значимый вывод должен ссылаться на конкретный входной артефакт (PR/конфиг/диаграмма/лог/тест).

## 2. Моделирование угроз и сценарии злоупотребления

Для полноценного процесса используйте [плейбук по моделированию угроз](../threat-modeling/playbook.ru.md). Он описывает выбор методологии, входные и выходные артефакты, рекомендуемый путь, сценарии атак, анализ риска, привязку контролей и упрощенный путь.

В рамках архитектурного ревью минимально требуется:
- обновить DFD/C4 или текстовое описание потоков данных и границ доверия;
- сформировать сценарии угроз для новых/измененных точек входа, потоков данных и привилегированных операций;
- сопоставить каждый сценарий с существующим контролем, разрывом или принятым риском;
- для `High`/`Critical` сценариев описать путь атаки и метод проверки;
- использовать STRIDE-LM только как легкую таксономию, а не как замену полноценного моделирования угроз.

Подтверждение:
- ссылка на threat model или таблицу сценариев угроз;
- ссылки на security-тесты, подтверждения контролей или результаты упражнений.

## 3. Ревью ключевых доменов безопасности

Проверяйте контроли через сценарии угроз и abuse cases из раздела 2: какой сценарий закрывает каждый контроль, где остается разрыв и чем подтверждена эффективность контроля.

### 3.1 Аутентификация

| Что проверить | Чем подтверждено |
|---|---|
| Механизмы: OAuth 2.0, OpenID Connect, API keys, mTLS | Конфигурация IdP/провайдера |
| Централизованная аутентификация; отсутствие самописной криптографии; валидация токена (`issuer`, `audience`, `expiration`) | Примеры валидаторов токенов; тесты отказа при невалидном токене |

### 3.2 Авторизация

| Что проверить | Чем подтверждено |
|---|---|
| Модель: RBAC/ABAC | Policy/role matrix |
| Контроль на каждом уровне (API/сервис/данные); отсутствие неявного доверия между сервисами; принцип наименьших привилегий | Примеры enforcement в коде/конфиге; тесты на запрет неразрешенных действий |

### 3.3 Аудит и журналирование

| Что проверить | Чем подтверждено |
|---|---|
| Логируются события аутентификации, решения авторизации, доступ к данным и критичные действия | Схема/конфиг логирования; примеры событий |
| Устойчивость к подмене; централизованный сбор; корреляция по `correlation_id` | Проверка retention/доступа к журналам; настройки неизменяемости/доступа |

## 4. Покрытие контролями

Для каждого пункта проверяйте связь со сценарием угроз, abuse case или явно зафиксированным требованием из раздела 2. Если связь неочевидна, фиксируйте это как разрыв или обоснованное исключение.

### 4.1 Безопасность данных

| Что проверить | Чем подтверждено |
|---|---|
| Классификация данных (Public/Internal/Confidential/Secret) | Data classification матрица |
| Шифрование при хранении; TLS при передаче | Конфиги шифрования/TLS |
| Управление секретами через Vault/KMS; риски утечки токенов/ключей | Политика ротации и доступов к секретам; результаты проверок secret scanning |

### 4.2 Безопасность интеграций

| Что проверить | Чем подтверждено |
|---|---|
| Валидация доверия внешних систем | Контракты API; схемы валидации; trust-конфиги |
| Валидация/санитизация входных данных | Правила валидации; негативные тест-кейсы |
| Безопасность исходящих вызовов (egress control) | Egress/policy конфиги |
| Retry/fallback без утечки чувствительных данных | Тесты отказоустойчивости и обработки ошибок интеграций |

### 4.3 Безопасность runtime

| Что проверить | Чем подтверждено |
|---|---|
| Контейнеры запускаются не от root | Манифесты/политики кластера |
| Удалены ненужные capabilities | Policy/manifest проверки; результаты сканов |
| Включены runtime-контроли (например, seccomp/AppArmor) | Конфиги runtime; runtime inspection |
| Секреты не хранятся в открытом виде в env/файлах | Результаты проверок образов/конфигов; secret scanning |

### 4.4 Соответствие требованиям

| Что проверить | Чем подтверждено |
|---|---|
| Источники требований (бизнес, безопасность, регуляторика) | Реестр требований |
| Подтверждение выполнения требований | Артефакты выполнения контролей; отчеты проверок |
| Выявленные конфликты и их разрешение | Протоколы согласований; decision log |
| Трассируемость `требование -> архитектурное решение -> контроль` | Traceability-матрица; подтверждения согласования со стейкхолдерами |

## 5. Лог решений, наблюдения и рекомендации

### 5.1 Лог решений и архитектурные заметки

Задокументируйте:
- допущения;
- компромиссы;
- обоснование решений;
- отклоненные альтернативы;
- принятые риски и технический долг с владельцем и трекером.

Исключения из контролей ведите по отдельной policy исключений безопасности команды (если такой документ есть, добавьте на него ссылку в ADR/decision log).

Подтверждение:
- ADR/decision log;
- ссылки на утвержденные исключения.

### 5.2 Наблюдения и рекомендации (обязательный формат)

Каждый finding оформляется строго по шаблону:
- ID:
- Наблюдение:
- Риск/влияние:
- Severity: `Low|Medium|High|Critical`
- Likelihood: `Low|Medium|High`
- Рекомендация (конкретное действие):
- Владелец:
- Due date:
- Метод проверки:
- Ссылки на подтверждения (PR/конфиг/лог/тест/диаграмма):

Требования:
- без двусмысленных формулировок;
- владелец и срок обязательны для всех `High`/`Critical`;
- метод проверки обязателен для закрытия finding.

## 6. Упрощенный путь для low-risk изменений

Используйте раздел 3.1 "Упрощенный путь" из [плейбука по моделированию угроз](../threat-modeling/playbook.ru.md#31-упрощенный-путь) как базу для минимального анализа.

Применяйте упрощенный путь только если одновременно верно:
- нет новых внешних интеграций;
- нет изменений в аутентификации/авторизации/сессиях;
- не обрабатываются новые чувствительные данные;
- нет новых internet-facing точек входа;
- нет расширения привилегий или границы доверия.

Минимальные обязательные проверки (6-8 пунктов):
- проверка аутентификации и авторизации для измененных компонентов;
- проверка валидации входа/выхода в точках интеграции;
- проверка секретов и TLS;
- проверка логирования критичных security-событий;
- проверка runtime hardening (не-root, capabilities, базовые runtime-контроли);
- фиксация findings в обязательном формате;
- финальный вердикт по gate-правилам.

Эскалация на полный ревью обязательна, если найден хотя бы один признак:
- потенциальный `High`/`Critical` риск;
- новая точка входа или новая граница доверия;
- сомнение в полноте подтверждений.

## 7. Финальный вердикт безопасности (правила gate)

Статусы:
- `Rejected`
- `Approved with risks`
- `Approved`

Обязательные правила:
- `Rejected`, если есть хотя бы один `Critical` без подтвержденной mitigation;
- `Approved with risks`, если есть принятые `High` при наличии владельца + срока + компенсирующих контролей;
- `Approved`, если открытые риски не выше согласованного порога (обычно не выше `Medium`) и для них есть план закрытия.

Дополнительно:
- обязательные исправления до релиза фиксируются отдельным списком;
- остаточные риски должны быть явно приняты уполномоченным владельцем.
