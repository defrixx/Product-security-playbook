# Чеклист ревью архитектуры безопасности

## 1. Область и контекст

Определите границы ревью:
- Тип изменения:
  - Новая система / сервис
  - Модификация существующей архитектуры
  - Внешняя интеграция
- Уровень области:
  - Приложение / Сервис
  - Платформа / Инфраструктура
  - Данные / Хранилище
- Границы доверия:
  - Внутренние
  - Внешние (third-party, internet-facing)

---

## 2. Анализ базового состояния vs изменений

Строгое сравнение: **до -> после**

- Добавленные компоненты
- Измененные компоненты
- Удаленные компоненты
- Изменения потоков данных:
  - Новые потоки данных
  - Измененные существующие потоки
- Изменения границ доверия

---

## 3. Идентификация рисков по триггерам

Определите триггеры рисков:
- Новые интеграции:
  - Внешние API
  - Партнерские системы
- Изменения в модели доступа:
  - Новые роли
  - Расширение привилегий
- Изменения в:
  - Аутентификации
  - Авторизации
  - Управлении сессиями
- Обработка чувствительных данных:
  - Введение ПДн / secrets / tokens
- Изменения модели развертывания:
  - Cloud / on-prem / hybrid

---

## 4. Анализ поверхности атаки

Оцените влияние на поверхность атаки:
- Новые точки входа:
  - API endpoints
  - UI
  - Фоновые задачи
- Экспозиция:
  - Доступно из интернета
  - Только внутренний доступ
- Новые протоколы / интерфейсы:
  - HTTP, gRPC, messaging
- Изменения в:
  - Сетевой экспозиции
  - Межсервисном взаимодействии

---

## 5. Анализ обхода защитных контролей

Определите возможности обхода:
- Можно ли обойти аутентификацию?
- Можно ли обойти авторизацию?
- Возможно ли повышение привилегий?
- Нарушения границ доверия
- Прямой доступ к backend без контролей
- Внутренние API, экспонированные наружу
- Теневые потоки:
  - Недокументированные пути доступа к данным

---

## 6. Ревью ключевых доменов безопасности

### 6.1 Аутентификация

- Механизмы:
  - OAuth 2.0
  - OpenID Connect
  - API keys / mTLS
- Проверки:
  - Централизованная аутентификация
  - Отсутствие самописной криптографии
  - Валидация токена:
    - issuer
    - audience
    - expiration

---

### 6.2 Авторизация

- Модель:
  - RBAC / ABAC
- Проверки:
  - Применение на каждом уровне (API, сервис)
  - Отсутствие неявного доверия между сервисами
  - Принцип наименьших привилегий

---

### 6.3 Аудит и логирование

- Что логируется:
  - События аутентификации
  - Решения авторизации
  - Доступ к данным
- Свойства:
  - Устойчивость к подмене
  - Централизованное логирование
- Трассируемость:
  - Correlation IDs

---

## 7. Безопасность данных

- Классификация данных:
  - Public / Internal / Confidential / Secret
- Хранение:
  - Шифрование at rest
- Передача:
  - Принудительное использование TLS
- Управление секретами:
  - Vault / KMS
- Риски утечки токенов

---

## 8. Безопасность интеграций

- Валидация доверия внешних систем
- Валидация / sanitization входных данных
- Исходящая безопасность:
  - Ограничение внешнего доступа
- Retry / fallback:
  - Без утечки чувствительных данных

---

## 9. Безопасность инфраструктуры и runtime

- Контейнеры:
  - Запуск не от root
  - Удаление ненужных capabilities
- Runtime-контроли:
  - seccomp / AppArmor
- Обращение с секретами:
  - Без хранения в открытом виде (env/files)

---

## 10. Threat Modeling / Abuse Cases

Применяйте STRIDE:

- Spoofing
- Tampering
- Repudiation
- Information Disclosure
- Denial of Service
- Elevation of Privilege

**Как применять:**

- Определите все **новые или измененные элементы**:
  - точки входа (API, UI, webhooks)
  - потоки данных (кто -> что -> куда)
  - интеграции (внутренние/внешние)

- Применяйте STRIDE **к каждому элементу и взаимодействию**, а не глобально:
  - где можно подменить идентичность
  - где можно модифицировать данные
  - где можно обойти контроль доступа

- Преобразуйте находки в **явные abuse cases**:
  - Abuse Case:
    <сценарий атаки>
  - Impact:
    <что скомпрометировано>
  - Mitigation:
    <необходимый контроль>

**Требования:**
- Как минимум один abuse case на каждую новую точку входа или поток данных
- Каждый abuse case должен сопоставляться с конкретным контролем или gap
- Отсутствие abuse cases указывает на неполный анализ

---

## 11. Требования compliance и стейкхолдеров

- Источники требований:
  - Бизнес
  - Безопасность
  - Регуляторные
- Валидация:
  - Требования выполнены
  - Конфликты выявлены
- Трассируемость:
  - Требование -> Архитектура -> Контроль

---

## 12. Наблюдения и рекомендации

Для каждого наблюдения:
- Finding:
  - Описание
  - Расположение
- Риск:
  - Влияние
  - Вероятность
- Recommendation:
  - Конкретное действие (без двусмысленности)

---

## 13. Лог решений / Архитектурные заметки

Задокументируйте:
- Допущения
- Компромиссы
- Обоснование решений
- Отклоненные альтернативы

**Требования:**
- Без двусмысленности
- Полная трассируемость решений
- Каждый принятый риск или технический долг ДОЛЖЕН иметь tracking ticket и назначенного владельца
- Для каждого исключения ДОЛЖНЫ быть определены:
  - expiry date
  - closure criteria
  - compensating controls
  - explicit approver

---

## 14. Финальный вердикт безопасности

- Статус:
  - Approved
  - Approved with risks
  - Rejected
- Условия:
  - Обязательные исправления до релиза
- Остаточные риски:
  - Явно принятые риски
