# Чеклист ревью seccomp в Kubernetes

## 1. Область и цель безопасности

Используйте этот чеклист, чтобы проверить, применяется ли seccomp для Kubernetes workload'ов **корректно, реалистично и безопасно**.

### Цель

Seccomp должен использоваться для:
- Снижения достижимой поверхности атаки ядра
- Блокировки явно опасных syscalls, которые позволяют breakout, escalation, stealth или обход policy
- Применения workload-specific baseline syscalls там, где это операционно реализуемо
- Дополнения более сильной изоляции и других контролей hardening workload'ов, а не их замены

### Не-цели

Seccomp **не** должен рассматриваться как:
- Полноценный sandbox
- Замена runtime isolation
- Замена удаления Linux capabilities
- Замена Pod Security контролей, разделения пользователей, AppArmor/SELinux или network policy
- Доказательство, что workload "secure", только потому что profile существует

---

## 2. Базовые вопросы перед ревью

Сначала подтвердите следующее:
- Seccomp вообще включен для workload?
- Profile применен на корректном уровне:
  - Pod-level
  - Container-level
- Profile custom, runtime default или auto-generated?
- Какой runtime используется:
  - Docker/Moby
  - containerd/runc
  - Другой
- Какие архитектуры должны быть покрыты:
  - x86_64
  - x86
  - x32
  - arm64
  - другие
- Какие Linux capabilities выданы контейнеру?
- Ожидается ли, что workload требует углубленного взаимодействия с ядром, или это обычный application process?

Это важно, потому что фактический эффект безопасности seccomp profile зависит от поведения runtime, покрытия архитектур и контекста capabilities, а не только от статического JSON/YAML, который вы ревьюите.

---

## 3. Принципы корректного дизайна

### 3.1 Предпочитайте "сначала блокировать опасное", а не "слепо блокировать все, что не наблюдали"

Дизайн seccomp должен основываться на:
- Сначала блокировке **известно опасных** syscalls
- Затем сокращении дополнительной поверхности syscalls там, где это обосновано
- Отказе от наивного предположения, что "неиспользуемый" автоматически означает "значимый для безопасности"

Ключевой момент: **ненужные syscalls** и **опасные syscalls** — не одно и то же. Блокировка опасных syscalls явно снижает риск. Блокировка произвольных неиспользуемых syscalls может только добавить хрупкости, если нет понятного снижения угрозы.

### 3.2 Применяйте по workload, а не как cargo-cult практику

Команда должна явно определить:
- Зачем seccomp используется для этого workload
- Какие классы атак снижаются
- Какие операционные компромиссы приемлемы
- Является ли seccomp вообще правильным инструментом для этого workload

### 3.3 Рассматривайте seccomp как один слой defense in depth

Используйте seccomp вместе с:
- Запуском не от root
- Удаленными capabilities
- `allowPrivilegeEscalation: false`
- Read-only filesystem, где возможно
- AppArmor/SELinux, где доступно
- Runtime default protections
- Admission policy
- Сетевыми контролями
- Более сильной изоляцией для high-risk workload'ов

## 4. Ревью источника и генерации profile

## 4.1 Если profile auto-generated, требуйте ручное ревью

Определите, был ли profile получен из:
- Tracing-based generation tools
- Security Profiles Operator workflows
- eBPF tracers
- ptrace/strace-like tooling
- OCI/runtime tracing

Если да, проверьте, что до одобрения было **ручное ревью syscalls**.

## 4.2 Не предполагайте, что tracing tools согласованы или полны

Ревьюеры должны учитывать следующее:
- Разные уровни tracing видят разные наборы syscalls
- Syscalls настройки runtime могут попасть в capture и быть ошибочно приняты за syscalls приложения
- Идентичный код может давать разные traces syscalls в зависимости от libc, ядра, сборки и условий runtime

## 4.3 Убедитесь, что trace захватил приложение, а не окружающий platform noise

Ищите загрязнение от:
- `containerd`
- `containerd-shim`
- `runc`
- init containers
- sidecars
- CNI components
- workflows инъекции секретов
- storage mounts
- самих tracing tools

---

## 5. Область применения: Pod vs Container

### 5.1 Убедитесь, что profile привязан на запланированном уровне

Проверьте, где применен profile:
- Pod security context
- Container security context

Затем подтвердите, что это намеренно.

### 5.2 Предпочитайте container-specific ревью, когда поведение внутри Pod различается

Будьте внимательны, когда:
- Есть init containers
- Есть sidecars
- У контейнеров разные обязанности
- Один контейнер более привилегирован, чем другой

Pod-wide profile может быть слишком широким и может смешивать startup/runtime поведение нескольких контейнеров.

## 6. Ревью опасных syscalls

Следующие syscalls должны рассматриваться как high-risk элементы ревью.

## 6.1 Tier 1: Default fail, если нет исключительной документированной причины

Эти syscalls должны быть **запрещены по умолчанию**:
- `bpf`
- `ptrace`
- `kexec_load`
- `init_module`
- `finit_module`

### Примечание ревью

Если любой из перечисленных выше разрешен, требуйте:
- Явное обоснование
- Security sign-off
- Компенсирующие контроли
- Ревью того, не должен ли workload вместо этого запускаться в более сильной изоляции

## 6.2 Tier 2: Значительный риск, требуется сильное обоснование

Тщательно проверяйте, если profile разрешает:
- `io_uring_setup`
- `io_uring_enter`
- `io_uring_register`
- `perf_event_open`
- `mount`
- `clone`
- `unshare`

## 6.3 Дополнительные чувствительные syscalls, явно выделенные в серии

Явно ревьюьте:
- `add_key`
- `keyctl`
- `userfaultfd`
- `clone3`
- `chroot`

---

## 7. Ревью io_uring: обязательный раздел

Если разрешен `io_uring_*`, рассматривайте это как крупное событие ревью.

### 7.1 Понимание ревьюером риска обхода

Убедитесь, что команда понимает: `io_uring` — это не просто оптимизация I/O. Он может работать как **syscall multiplexer**, позволяя операции, которые иначе были бы заблокированы на уровне syscall.

`io_uring_enter()` может отправлять операции, соответствующие поведению network, file, filesystem, polling, splice, futex/wait, ioctl и xattr, без прямого вызова заблокированных syscalls.

### 7.2 Проверьте конкретный anti-pattern

Считайте серьезным anti-pattern следующее:
- Блокируются `socket`, `connect`, `send`, `recv`
- Но разрешены `io_uring_setup` + `io_uring_enter`

Это создает ложное ощущение безопасности. Сетевые ограничения можно обойти таким способом, потому что заблокированные сетевые syscalls не вызываются напрямую.

### 7.3 Проверьте утечку опциональной функциональности

Проверьте, что `io_uring` не включен только потому, что приложение opportunistically проверило его наличие.

### 7.4 Если `io_uring` оставлен, документируйте почему

Требуйте документацию о:
- Почему это нужно
- Есть ли fallback
- Какая поверхность атаки принимается
- Какие компенсирующие контроли существуют

---

## 8. Ревью eBPF / bpf: обязательный раздел

Если разрешен `bpf`, рассматривайте profile как предположительно небезопасный, пока не доказано обратное.

### 8.1 Подтвердите, что `bpf` действительно требуется

`bpf` — один из первых syscalls, на который смотрит атакующий, потому что он позволяет мощное взаимодействие с ядром и связан со сценариями escalation и stealth.

### 8.2 Проверьте скрытое происхождение `bpf` в profile

Проверьте, попал ли `bpf` в profile из-за:
- eBPF-based tracing tools
- поведения runtime/CNI
- шумного Pod-level profiling
- выданных capabilities
- несвязанных инфраструктурных компонентов

### 8.3 Считайте `bpf` несовместимым с большинством обычных application workload'ов

Для типовых бизнес-workload'ов разрешение `bpf` обычно следует считать провалом дизайна, если нет очень сильного и явного platform-level обоснования.

---

## 9. Capabilities и seccomp нужно ревьюить вместе

### 9.1 Не ревьюйте seccomp в изоляции

Проверьте выданные контейнеру capabilities.
Capabilities могут существенно влиять на эффективную seccomp policy. В Docker/Moby и containerd-флоу обработка, чувствительная к capabilities, может менять разрешения, включая поведение syscalls вокруг `bpf` и других чувствительных функций.

### 9.2 Отмечайте опасные комбинации capabilities

Требуйте ревью, когда seccomp сочетается с capabilities, такими как:
- `CAP_SYS_ADMIN`
- `CAP_BPF`
- другими мощными capabilities, взаимодействующими с ядром

### 9.3 Anti-pattern

Считайте anti-pattern следующую логику:
- "У нас есть seccomp, значит дополнительные capabilities допустимы"

Это неверное рассуждение. Capabilities могут резко снизить ценность seccomp.

---

## 10. Покрытие архитектур и ABI

### 10.1 Подтвердите, что покрытие архитектур задано явно

Проверьте, что profile покрывает архитектуры, релевантные развертыванию.

### 10.2 Проверяйте слепые зоны x32 ABI, где это релевантно

Если `SCMP_ARCH_X32` не объявлен, заблокированные номера syscalls в некоторых условиях могут быть достижимы через пути x32 ABI.

### 10.3 Не предполагайте, что один profile ведет себя одинаково во всех средах

Разные версии ядра, варианты libc, toolchains и runtime могут менять набор syscalls, которые реально используются или наблюдаются. Один и тот же код может легитимно выдавать разные списки syscalls в разных системах.

---

## 11. Ревью комбинаций и обходов

Хорошее ревью seccomp должно проверять **комбинации**, а не только отдельные строки syscalls.

### 11.1 Обязательные проверки комбинаций

Проверяйте такие паттерны:
- `io_uring_setup` + `io_uring_enter`, пока сетевые syscalls заблокированы
- `io_uring_setup` + `io_uring_enter`, пока file I/O syscalls заблокированы
- `io_uring_setup` + `io_uring_enter`, пока syscalls путей файловой системы заблокированы
- `io_uring_setup` + `io_uring_enter`, пока syscalls polling заблокированы
- `io_uring_setup` + `io_uring_enter`, пока заблокированы `splice`/`tee`/`vmsplice`
- `io_uring_setup` + `io_uring_enter` при ограничениях process-wait/futex
- `io_uring_setup` + `io_uring_enter`, когда `ioctl` заблокирован
- `io_uring_setup` + `io_uring_enter`, когда syscalls xattr заблокированы

Эти классы комбинаций показывают, почему построчное ревью syscalls может пропустить реальные пути обхода.

### 11.2 Anti-pattern

Считайте anti-pattern следующим:
- "Мы заблокировали классические имена syscalls, следовательно функция заблокирована"

Этого недостаточно, когда существуют механизмы мультиплексирования.

---

## 12. Ревью операционной корректности

### 12.1 Проверьте, что необходимые syscalls остаются доступными

Profile, который ломает production, — плохой profile.
Отсутствие нужных syscalls приводит к падениям и сбоям развертывания, и это самый непосредственный режим операционного отказа.

### 12.2 Избегайте добавления опасных syscalls только ради старта workload

Это часто хуже, чем сбой старта, потому что создает работающий, но менее безопасный workload.
Profile может пройти CI и успешно развернуться, одновременно ослабляя безопасность, если разрешает опасные syscalls.

### 12.3 Требуйте тестовое покрытие в реалистичном runtime-поведении

Подтвердите, что profiling и validation выполнялись с:
- Реальным startup path
- Реальной инициализацией зависимостей
- Реальными sidecars/init containers, если применимо
- Поведением kernel/runtime, похожим на production
- Релевантными условиями архитектуры и libc

---

## 13. Осведомленность о runtime и реализации

### 13.1 Подтвердите, как runtime формирует эффективный profile

Например:
- Profile загружается как статический JSON?
- Он генерируется динамически?
- Применяются вариации на основе capabilities при старте?

Containerd может динамически генерировать эффективную seccomp-конфигурацию в зависимости от capabilities процесса, поэтому реальную применяемую policy сложнее аудировать только по статическим файлам.

### 13.2 Anti-pattern

Считайте anti-pattern следующим:
- "YAML выглядит безопасно, значит runtime-поведение безопасно"

Реальный filter, примененный во время runtime, может отличаться по эффекту в зависимости от деталей реализации, capabilities и контекста платформы.

## 14. CI/CD и управление жизненным циклом

### 14.1 Ревью seccomp не должно быть разовой активностью

Переоценивайте seccomp каждый раз, когда меняется что-либо из следующего:
- Код приложения
- Зависимости
- Base image
- libc
- Kernel
- Тип node
- Container runtime
- Capabilities
- Sidecars/init containers
- Поведение workload

Seccomp — не разовое упражнение; его нужно интегрировать в CI, потому что даже обычные изменения могут инвалидировать предыдущие допущения.

### 14.2 Добавьте policy gates в CI

Рекомендуемые контроли:
- Падайте build, если присутствуют запрещенные syscalls
- Падайте build, если присутствуют опасные комбинации
- Сравнивайте custom profile с runtime default
- Требуйте ручное security review для high-risk deltas
- Ведите исключения с expiry и owner

### 14.3 Отслеживайте дрейф profile

Команда должна уметь ответить:
- Какая версия profile развернута
- Какой версии image/workload она соответствует
- Когда последний раз выполнялся profiling
- Что изменилось с момента последнего одобрения

### 14.4 Обязательная проверка effective-profile на nodes

Не полагайтесь только на YAML/JSON в репозитории. Проверяйте, что реально применяется на running nodes.
Минимальные контроли:
- В CI/CD: сохраняйте hash profile и ожидаемый источник profile для каждой версии workload.
- В runtime: сэмплируйте running pods в каждом node pool и проверяйте effective seccomp config через runtime inspection (`crictl inspect` / runtime API) не реже чем каждые `24h`.
- Триггерите немедленную проверку после любого изменения:
  - node image
  - версия kernel
  - версия/config container runtime
  - выданные capabilities
- Алерт и блокировка promotion, когда hash runtime profile отличается от одобренного hash без одобренного исключения.

### 14.5 Управление исключениями для опасных syscalls/комбинаций

Каждое исключение (разрешенный опасный syscall или отмеченная комбинация) должно включать:
- Owner
- Tracking ticket
- Явное заявление о риске
- Компенсирующие контроли
- Дату истечения (по умолчанию `14d`, жесткий максимум `45d`)
- План удаления и шаг верификации

Истекшие исключения должны приводить к fail CI policy gates.

## 15. Рекомендуемые эвристики ревью

Используйте эти практические эвристики ревью.

### 15.1 Хорошие признаки

- Назначение profile задокументировано
- Опасные syscalls явно ревьюятся
- Capabilities минимизированы
- Profile container-specific там, где это нужно
- Покрытие архитектур задано явно
- Auto-generated результат вручную курирован
- `io_uring` и `bpf` проходят особое scrutiny
- CI непрерывно валидирует profile
- Команда сравнивает с runtime defaults
- Команда понимает, что seccomp — только один слой

### 15.2 Плохие признаки

- Profile существует только потому, что "так говорит best practice"
- Profile auto-generated и никогда вручную не ревьюился
- Ревью сфокусировано только на "количестве заблокированных syscalls"
- Pod-level profiling захватил несвязанные компоненты
- Опасные syscalls были включены потому, что "с ними приложение стартовало"
- Разрешен `io_uring`, пока network/file syscalls заблокированы
- `bpf` присутствует в profile обычного приложения
- Capabilities выданы без повторного ревью seccomp
- Покрытие architecture/ABI было проигнорировано
- После изменений не выполняется ретест

## 16. Частые anti-patterns

### Anti-pattern 1: Auto-generated profile = approved profile

Auto-generation — это только входные данные, а не решение по безопасности.

### Anti-pattern 2: Подсчет заблокированных syscalls как основная метрика успеха

Более строгий profile не становится автоматически безопаснее, если опасные syscalls все еще разрешены.

### Anti-pattern 3: Блокировка классических сетевых syscalls при открытом `io_uring`

Это может оставить сетевой путь фактически достижимым.

### Anti-pattern 4: Ревью статического YAML/JSON без runtime-контекста

Runtime-поведение, capabilities и детали реализации по-прежнему важны.

### Anti-pattern 5: Смешивание syscalls приложения с syscalls runtime/init/CNI

Это может расширить profile и ввести опасные разрешения.

### Anti-pattern 6: Сохранение опасных syscalls, потому что "workload с ними работает"

Успешный запуск workload не означает, что profile приемлемо безопасен.

## 17. Минимальный фреймворк решений ревьюера

Используйте этот упрощенный decision flow.

### Fail immediately if:

- Разрешен `bpf` без исключительного обоснования
- Разрешен `ptrace` без исключительного обоснования
- Разрешены `kexec_load`, `init_module` или `finit_module` без исключительного обоснования
- Разрешен `io_uring`, но последствия обхода не были ревьюированы
- Capabilities и seccomp ревьюились раздельно
- Источник profile — auto-generated и некурированный
- Эффективное runtime-поведение неизвестно

### Escalate for manual security review if:

- Присутствует `io_uring_*`
- Присутствуют `mount`, `unshare`, `clone`, `clone3`, `perf_event_open`, `userfaultfd`, `keyctl` или `add_key`
- Profile Pod-wide при наличии нескольких контейнеров
- Workload имеет повышенные capabilities
- Runtime динамически мутирует effective policy
- Workload требует более сильных гарантий изоляции, чем seccomp может реалистично обеспечить

### Accept with conditions if:

- Опасные syscalls удалены или строго обоснованы
- Scope корректен
- Покрытие архитектур задано явно
- Capabilities минимизированы
- Оценены комбинации обхода
- CI непрерывно валидирует profile
- Команда понимает и документирует остаточный риск

---

## 18. Финальное заявление ревью

Seccomp profile должен считаться **хорошим** только когда он:
- Снижает реальную поверхность атаки
- Исключает известно опасные syscalls, если нет явного обоснования
- Избегает очевидных комбинаций обхода
- Соответствует реальному workload, а не шуму runtime
- Ревьюится вместе с capabilities, runtime и контекстом архитектуры
- Поддерживается непрерывно, а не создается один раз и забывается

Profile, который просто строгий, auto-generated или присутствует в YAML, недостаточен.
